Implementation & Technologies

Single page application: The html is loaded on the client side (web browser) only once when application initialized, after that all user interfaces are generated by the client itself. Client application build upon Ural SPA Framework on coffee-script.

Pros: Agile user interface, no page reloading needed. Reduction in size of the data passed from server to client (only raw data, not HTML). Since this is a browser application, it could work on any device: mobile phones, tablets, all desktop systems and because of the agile user interface it could be felt like native application created specially for device where it launched.

Cons: The interface itself could work only in modern browsers, the client machines also need to be not too obsolete.

Data Service

Data service is implemented using OData service protocol . OData protocol allow works with data in RESTFUL way, provide any client the ability to consume service interfaces in easy way. The protocol well definition and documentation guarantee that developers won't have problems with learning and understanding how to access service data.

Data Layer

Data stored in Mongo database . Among its features: no data schemes (reduction in development time), low cost of storage, intuitive admin interface (all java script). The minuses are: consume large disk space to store data (in general space of the disk available should be 2x more than the data stored), indexing should be done in "right" way, large amount of RAM memory needed to operate with large volumes of data, though these drawbacks could be applied to any modern data base.
